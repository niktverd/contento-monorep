# Fast-Track Architecture: Modular Monolith (Next.js + NestJS)

## Overview

The goal is to launch the Contento system in a few weeks while maintaining the logical boundaries of the architecture defined in `01.init.md`. To achieve this, we will pivot to a **Modular Monolith** approach. 

Instead of building, deploying, and networking 5 separate microservices (Publish API, Content Factory API, Spam API, Contario UI, Admin UI), we will consolidate them into **two deployable units** while strictly separating their code inside the repository.

1.  **`apps/core-api` (NestJS)**: A single backend application containing `publish`, `content-factory`, and `spam` as distinct modules. It handles all HTTP requests, Temporal workflows, and database interactions.
2.  **`apps/contario` (Next.js)**: A single frontend application serving both end-users (B2C) and internal admins (Spam, Publish, Content Factory).

│   │   └── package.json
│   │

│       ├── src/
│       │   ├── app/

## 🚦 Strict Rules for "Modular" Monolith

To ensure we can split this into microservices later without rewriting everything, we must enforce these rules:

1.  **NO Database Cross-Talk**: The `SpamModule` cannot query `Publish` database tables using Prisma. It must call a method exposed by the `PublishService`.
2.  **Use Contracts**: The API boundary (REST controllers) between the UI and the Backend *must* use the shared Zod schemas from `shared/` for strict typing. No GraphQL.
3.  **Temporal is Centralized**: `core-api` starts the Temporal worker. Workflows are isolated in their respective folders (`publish/workflows.ts`, `content-factory/workflows.ts`).
4.  **No `Spam` to `Network` calls**: `SpamModule` can never talk to Instagram directly. It must call `PublishService.schedulePost(...)`.

## 🛠️ Step-by-Step Implementation Plan

### Phase 1: Infrastructure & DB Consolidation (Days 1-2)

**Goal:** Get a single local environment running with Prisma.

1.  **Spin up `docker-compose.yml`**:
    *   1x PostgreSQL (Single database named `contento_core`).
    *   1x Temporal Cluster (Server, UI, Cassandra/Postgres for Temporal state).
    *   1x Keycloak.
2.  **Initialize unified Prisma Schema**:
    *   Delete the old databases/Prisma setups from the legacy repos.
    *   Create `apps/core-api/prisma/schema.prisma`.
    *   Combine schemas: `User`, `Account`, `Organization`, `Post`, `Scenario`, `Campaign`. Add logical namespaces using `@map` or standard prefixes if needed (e.g., `publish_Posts`, `spam_Campaigns`).

### Phase 2: Building the `core-api` (NestJS) (Days 3-7)

**Goal:** Create a unified REST API and start Temporal.

1.  **Scaffold `apps/core-api`**: Use Nest CLI to create the base project.
2.  **Setup the `shared` package**: Extract the Zod request/response schemas from `01.init.md` into the `./shared` folder. Ensure both `core-api` and `contario` can import them.
3.  **Implement `AuthModule`**: Integrate with Keycloak to validate JWTs locally.
4.  **Implement `PublishModule`**:
    *   *Controller*: `POST /api/publish/now` (Validates against Zod contract).
    *   *Temporal*: Define `PublishActivities` (call Instagram Graph API) and `PublishWorkflows` (handles retries, rate limits).
5.  **Implement `ContentFactoryModule`**:
    *   *Controller*: `POST /api/content/generate`
    *   *Temporal*: Define heavy `RenderVideoActivity` (runs FFmpeg locally or triggers a separate script if needed).
6.  **Implement `SpamModule`**:
    *   *Controller*: `POST /api/spam/campaign`
    *   *Service logic*: Calls methods from `ContentFactoryModule` to generate videos, then calls `PublishModule` to schedule them.

### Phase 3: Building `contario` (Next.js) (Days 8-12)

**Goal:** One frontend to rule them all.

1.  **Scaffold `apps/contario`**: Base Next.js App Router project with Tailwind CSS.
2.  **Admin UI (`/admin/publish`)**: Forms to connect Instagram accounts and view the Temporal queue. Connects to `core-api/api/publish`.
3.  **Admin UI (`/admin/content`)**: Flow to define FFmpeg scenarios and upload templates.
4.  **Admin UI (`/admin/spam`)**: Dashboard to create campaigns and track success.
5.  **B2C UI (`/`)**: The public-facing flow where regular users interact with standard content generation workflows.

### Phase 4: CI/CD & Deployment (Days 13-14)

**Goal:** Ship it.

1.  **Hetzner Server Setup**: Provision a dedicated root server or a high-powered cloud instance (e.g., CPX/CCX line) on Hetzner to ensure you have enough CPU/RAM for FFmpeg processing.
2.  **Install Dokploy**: Install Dokploy (the open-source Vercel/Heroku alternative) on the Hetzner server. It natively supports monorepos, Dockerfiles, and zero-downtime deployments.
3.  **Frontend (`apps/contario`)**: Add it to Dokploy as a Next.js application, pointing the build directory to the `apps/contario` folder.
4.  **Backend (`apps/core-api`)**: Deploy it via Dokploy using a standard Dockerfile or Nixpacks. Ensure the server has FFmpeg installed (if relying on host binaries) or bake FFmpeg into the NestJS Docker image.
5.  **Managed/Self-Hosted Services**: Use Dokploy to quickly compose the necessary databases: PostgreSQL (with persistent volumes), Redis (if required for cache), Keycloak, and Temporal.

## 📈 Why this works

*   **Fastest Time to Market:** You skip building and configuring 6 different repos, Dockerfiles, and CI pipelines.
*   **Easy to Reason About:** If a video fails to publish, you look in exactly one project (`core-api`) instead of tracing HTTP calls across 3 microservices.
*   **Zero Migration Regrets:** By strictly keeping the code inside `core-api` separated into NestJS Modules (`publish`, `content-factory`, `spam`), lifting them out into true microservices next year will literally be as simple as copying the folder into a new repo.

# App Architecture — Contento Monorepo

## Overview

The system consists of **3 backend services** (each with its own admin/control UI), **1 B2C client app**, an **auth layer** (Keycloak), and **shared contract packages** (Zod schemas).

Every service is **independent** and communicates with others **only via HTTP API**, validated by shared Zod schemas.

---

## Services

### 1. PUBLISH SERVICE (Temporal-based)
Manages scheduling and publishing content to social networks.

**Responsibilities:**
- Управление аккаунтами в социальных сетях
- Управление аккаунтами клиентов (организации, права доступа)
- Планирование публикации контента (очередь с интервалом)
- Публикация контента по запросу или расписанию
- Оркестрация через Temporal

**Apps:**
- `publish-api` — backend (Express + Temporal, DB: publish_db)
- `publish-ui` — admin UI для управления публикациями и аккаунтами

---

### 2. CONTENT FACTORY
Creates content: video, images, text. Operates independently.

**Responsibilities:**
- Сценарии производства видео / картинок / текста
- Привязка к источникам входных данных через API
- Управление пайплайнами создания контента

**Apps:**
- `content-factory-api` — backend (Express, DB: content_factory_db)
- `content-factory-ui` — UI для управления сценариями и пайплайнами

---

### 3. SPAM SYSTEM
Automates mass-distribution of content. **Consumes APIs** of Publish Service and Content Factory — does NOT duplicate their logic.

**Responsibilities:**
- Управление стратегиями массовой рассылки
- Настройка сценариев обработки входящих видео
- Использует Content Factory для создания контента
- Использует Publish Service для публикации

**Apps:**
- `spam-api` — backend (Express: потребляет publish-api и content-factory-api, DB: spam_db)
- `spam-ui` — UI для управления спам-кампаниями

---

### 4. CONTARIO (B2C Client)
Public-facing client application for end users.

**App:**
- `contario` — Next.js frontend

---

### 5. AUTH — Keycloak
Centralized authentication and authorization for all services and UIs.

## Shared Contracts (Zod)

Contract packages are **npm workspace packages** — they are NOT published to npm registry.
They are resolved locally via `"workspaces"` in root `package.json`.

During **Docker build**, contracts are included by copying the relevant `packages/` into the Docker context:

```dockerfile
# Example: Dockerfile for spam-api
COPY package.json package-lock.json ./
COPY packages/contracts-publish ./packages/contracts-publish
COPY packages/contracts-content-factory ./packages/contracts-content-factory
COPY packages/contracts-spam ./packages/contracts-spam
COPY packages/shared-utils ./packages/shared-utils
COPY apps/spam-api ./apps/spam-api

RUN npm ci --workspace=apps/spam-api
```

Each contract package exports Zod schemas + inferred TypeScript types:

```typescript
// packages/contracts-publish/src/publish-request.ts
import { z } from 'zod';

export const PublishRequestSchema = z.object({
  accountId: z.string().uuid(),
  contentId: z.string().uuid(),
  scheduledAt: z.string().datetime().optional(),
  platform: z.enum(['instagram', 'tiktok', 'youtube']),
});

export type PublishRequest = z.infer<typeof PublishRequestSchema>;

// Used by publish-api to validate incoming requests
// Used by spam-api to construct valid outgoing requests
```

## Access Control (Keycloak)

| Service | Roles |
|---|---|
| publish-api / publish-ui | `publish-admin`, `publish-viewer` |
| content-factory-api / content-factory-ui | `content-admin`, `content-viewer` |
| spam-api / spam-ui | `spam-admin` |
| contario | `user` (end-user) |

Each service validates JWT tokens issued by Keycloak. Service-to-service calls use **client credentials** grant.

---

## Key Principles

1. **Independence** — each app is a standalone deployable unit
2. **API-only integration** — no direct code imports between apps
3. **Contract-first** — all inter-service interfaces defined by Zod schemas in `packages/`
4. **Workspace packages** — contracts are local npm packages, not published; bundled into Docker at build time
5. **Centralized auth** — Keycloak manages all access control
6. **Directed dependencies** — Spam → (Publish, Content Factory). No cycles.
